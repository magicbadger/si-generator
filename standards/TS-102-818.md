# ETSI TS 102 818: XML Specification for Service and Programme Information (SPI)
**Version**: 3.5.1  **Date**: 2023-12  **Status**: Normative
**Source**: https://www.etsi.org/deliver/etsi_ts/102800_102899/102818/03.05.01_60/ts_102818v030501p.pdf

## Scope
Defines the XML schema data model for Service and Programme Information (SPI) and the methods for its delivery over DAB/DRM broadcast and IP networks. The format is used both to transmit service/schedule data to SPI client applications on devices, and as an interchange format between broadcasters, service providers, network operators and content providers. Does not define binary encoding (that is covered by ETSI TS 102 371).

## Key Concepts
- **SPI**: Service and Programme Information — the collective data model defined by this standard
- **SI (Service Information)**: Metadata describing radio services, their bearers, logos, genres, links and groupings
- **PI (Programme Information)**: Schedule data — linear and on-demand programme listings with timings and bearer references
- **GI (Group Information)**: Grouping of programmes/programme events into series, shows, themes
- **bearer**: The method of carriage (DAB, DRM, FM, IP stream) identified by a bearerURI per ETSI TS 103 270
- **CRID**: Content Reference ID (TV-Anytime) used as the globally unique identifier for programmes, programme events, programme groups and memberOf groupings. Format: `crid://<authority>/<data>`
- **shortCRID**: 24-bit integer (0–16 777 215) used as a compact local identifier alongside CRID; unique within an SPI service for a minimum of 6 months
- **MOT**: Multimedia Object Transfer protocol, used for broadcast delivery of XML and media assets
- **RadioDNS**: DNS-based mechanism for hybrid radio application discovery (ETSI TS 103 270)
- **service following**: Automatic switching between broadcast and IP bearers of the same service
- **presentationLanguage**: Language(s) used in the audio content (distinct from `xml:lang` which covers metadata language)

### service element
| Attribute | Type | M/O | Notes |
|-----------|------|-----|-------|

- Broadcast: PNG; sizes 32x32, 112x32, 128x128, 320x240 (in MOT carousel)

### programmeEvent element
Same attributes and structure as programme, except: no child `programmeEvent` nesting; requires >=1 location; onDemand is 0+.

### location element
Children (in order): time or relativeTime (1+), bearer (0+).
No bearer = inherit from parent programme or service.

### time element
| Attribute | Type | M/O | Notes |
|-----------|------|-----|-------|
| time | timePointType | M | Billed start |
| duration | durationType | M | Billed duration |
| actualTime | timePointType | O | Actual start if differs |
| actualDuration | durationType | O | Actual duration if differs |

Multiple `time` elements = repeat broadcasts of identical content within document scope.

### relativeTime element
Same attributes as time but all durations (offset from programme start); used within programmeEvent only.

### onDemand element
Children (in order): presentationTime (exactly 1), acquisitionTime (0+), bearer (1+).

|-----------|------|-----|-------|

## Rules and Constraints

1. [M] All XML documents SHALL use UTF-8 character encoding (ISO/IEC 10646).
2. [M] Reserved XML characters in text SHALL be encoded as entity references or CDATA sections.
3. [M] shortCRID SHALL be unique within an SPI service and SHALL NOT be reused within that service for a minimum of 6 months.
4. [M] CRID authority SHALL be a registered Internet domain name the author has permission to use.
5. [M] Duration values SHALL include the "PT" designator and at least one number+designator.
6. [O] Durations > 18 hours SHOULD be avoided for broadcast binary encoding compatibility.
7. [M] service element SHALL include at least one shortName and one mediumName in the default document language.
8. [M] service element SHALL include at least one bearer or radiodns element.
9. [M] programme and programmeEvent SHALL include at least one mediumName in default document language.
10. [M] programme SHALL include at least one location or onDemand element.
11. [M] programmeEvent SHALL include at least one location element.
12. [M] onDemand SHALL contain exactly one presentationTime and at least one bearer.
13. [M] For DAB bearers: mimeValue SHALL be `audio/mpeg` (DAB) or `audio/aacp` (DAB+).
14. [M] For DRM bearers: mimeValue SHALL be the MIME type of the encoded audio stream.
15. [M] For IP bearers: mimeValue is required.
16. [M] serviceInformation SHALL NOT define both serviceProvider attribute and serviceProvider child element.
17. [M] logo_colour_square: PNG 32x32 at 256 colour depth; SHALL NOT use mimeValue/width/height attributes.
18. [M] logo_colour_rectangle: PNG 112x32 at 256 colour depth; SHALL NOT use mimeValue/width/height attributes.
19. [M] logo_unrestricted SHALL include mimeValue, width, height attributes.
20. [M] Service logos for broadcast delivery SHALL provide 4 required sizes (32x32, 112x32, 128x128, 320x240) in MOT carousel.
21. [M] Service logos for IP delivery SHALL provide 5 required sizes (32x32, 112x32, 128x128, 320x240, 600x600) at HTTP URLs.
22. [M] geolocation with ref attribute SHALL NOT define child elements.
23. [M] geolocation allow attribute SHALL only be present on children of streaming bearer elements.
24. [M] polygon for streaming bearer SHALL contain minimum 4 and maximum 100 coordinate pairs; first and last pairs shall be identical (closed polygon).
25. [M] For a given streaming bearer, total polygon coordinate pairs across all child geolocation elements SHALL NOT exceed 100.
26. [M] Geolocation polygon/country boundaries for streaming bearers with allow attribute SHALL NOT touch or intersect each other.
27. [M] A receiver that does not know its location SHALL NOT use a bearer with any child geolocation element with allow=false.
28. [M] IP clients SHALL correctly follow HTTP redirects; SHALL respect indicated document expiry; SHOULD cache documents.
29. [M] Clients SHALL NOT present client identification when requesting from non-TLS application servers.
30. [M] radiodns serviceIdentifier: 1-16 lowercase alphanumeric chars [a-z0-9]; unique across services using the same FQDN.
31. [M] Broadcast bearer validation: for each broadcast bearer in an SI document found by IP, the client SHALL validate via DNS RadioDNS lookup before use.
32. [M] Devices SHALL switch away from a geolocation-restricted IP bearer within 60 seconds of entering a denied area.
33. [M] alias and phoneme SHALL be used only for voice recognition/TTS improvement; SHALL NOT be substitutes for keywords or genres; SHALL NOT represent other services/programmes.
34. [M] Only one alias with prefer=true allowed per language within a parent element.
35. [M] Only one phoneme with prefer=true allowed per alphabet+language within a parent element.
36. [O] Bitrate attribute SHOULD be set for constant-bitrate IP streams; SHOULD NOT be set for variable-bitrate.
37. [O] All available bearers SHOULD be listed in the service element to support service following.
38. [O] It is strongly recommended that service providers implement at least one TLS (HTTPS) application server.
39. [O] Clients SHOULD prefer TLS application servers over non-TLS.
40. [M] Multiple time elements within a location SHALL only be used when the programme is identical at all repeat times.

## Workflows and Sequences

### SI Document Discovery over IP
1. [M] Method 1 — DNS SRV lookup:
   a. Determine Authoritative FQDN via RadioDNS lookup (TS 103 270).
   b. Query `_radiospi._tcp.<fqdn>` (TLS); if found, SI URL = `https://<host>:<port>/radiodns/spi/3.1/SI.xml`.
   c. Otherwise query `_radioepg._tcp.<fqdn>`; if found, SI URL = `http://<host>:<port>/radiodns/spi/3.1/SI.xml`.
2. [O] Method 2 — HTML `<link rel="radioepg">` in service website header.
3. [O] Method 3 — Fixed path on service website: `<scheme>://<host>:<port>/radiodns/spi/3.1/SI.xml`.

### PI Document Discovery over IP
1. Discover Authoritative FQDN (from RadioDNS lookup or from `radiodns` element in matched SI).
2. Perform DNS SRV lookup for SPI application (see above).
3. Construct URL: `https://<host>:<port>/radiodns/spi/3.1/<ServiceIdentifier>/YYYYMMDD_PI.xml`.
4. If document does not cover desired time window, request adjacent dates.
5. On-demand PI documents linked via `<link mimeValue="application/xml+pi">`.

### GI Document Discovery over IP
- GI documents linked via `<link mimeValue="application/xml+gi">` from SI, PI or GI documents.

### Broadcast Delivery (DAB/DRM)
1. XML is binary-encoded per ETSI TS 102 371 and carried in MOT directory mode.
2. Document naming conventions:
   - SI: `YYYYMMDD_<name>_SI.xml` (one per ensemble/channel)
   - PI: `YYYYMMDD_<service>_PI.xml` (one per service per day; programmes ordered by start time)
   - GI: `YYYYMMDD_<name>_GI.xml` (one per ensemble/channel)
3. Multimedia assets (logos) carried in same MOT carousel; MOT ContentName matches `multimedia/@url`.
4. For DAB Filecast on-demand: each audio file named `<shortId>.mp4a` in MOT carousel; no `location` elements in PI; `onDemand` element is mandatory.

### Broadcast Bearer Validation (IP-acquired SI)
1. For each broadcast bearer in the SI, perform RadioDNS lookup to get its Authoritative FQDN.
2. [M] If FQDN matches the SI document's FQDN -> bearer is trusted.
3. [M] Otherwise perform DNS lookup of application SRV records for that FQDN; if returned host matches SI document host -> trusted.
4. [M] Otherwise -> untrusted; SHALL NOT be used.

### Service Following (Hybrid Devices)
1. Initial bearer selection considers: device/user preference, bearer cost (lower = preferred), geolocation restrictions.
2. Start from most preferred bearer; try each in turn until successful reception.
3. Monitor broadcast quality continuously; switch to IP if all broadcast options exhausted.
4. Switch back to broadcast when available with equal or lower cost.
5. Bearer offset (ms) may assist co-timing during bearer switches.
6. SI-defined bearers used unless PI provides programme-level bearer overrides (see Table E.1).

## Conformance Requirements
- Document XML namespace: `http://www.worlddab.org/schemas/spi`
- Schema version 3.5: `spi_35.xsd` (available at `https://www.worlddab.org/schemas/spi/spi_35.xsd`); XSD 1.1 processor required
- SI document: root element `serviceInformation`; services and serviceGroups as children
- PI document: root element `epg`; schedule child with scope and programme elements
- GI document: root element `epg`; programmeGroups child
- Character encoding: UTF-8 only
- Service logos: mandatory 4 sizes for broadcast, 5 sizes for IP
- Minimum name elements: shortName + mediumName for services; mediumName for programmes/groups
- All CRID identifiers: globally unique; shortCRID: unique within SPI service, not reused for 6+ months
- DNS SRV service names: `_radiospi._tcp` (TLS) and `_radioepg._tcp` (non-TLS)
- HTTP behaviour: follow redirects, respect cache headers, support compression (Accept-Encoding/Content-Encoding)
- Schema extensible via foreign namespaces (xs:anyAttribute and xs:any with processContents="lax" at defined extension points)

## Summary Notes
- The standard supersedes the broadcast-only ETSI TS 102 818 V1.5.1 EPG specification; the `ensemble` element was replaced by `services`/`serviceGroups` for hybrid use; binary encoding remains compatible via TS 102 371 V3.1.1+.
- The SPI XML schema permits extension via foreign namespaces at all major element types (`xs:any` with `processContents="lax"`), allowing platform-specific metadata to be added without breaking core validation.
- Annex G provides a PTy->TV-Anytime genre mapping table for DAB/DRM programme type codes.
- Annex D provides detailed logo size guidance and HTTP content negotiation for extended logo implementations.
- Annex E provides detailed service-following implementation guidance including a decision matrix for bearer switching.
- V3.5.1 additions over V3.4.1: `creationTime` attribute on multimedia (for logo version control) and `presentationLanguage` element for services, schedules, programmes and programme events.
