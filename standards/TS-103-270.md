# ETSI TS 103 270: RadioDNS Hybrid Radio; Hybrid lookup for radio services
**Version**: 1.4.1  **Date**: 2022-05  **Status**: Normative  
**Source**: https://www.etsi.org/deliver/etsi_ts/103200_103299/103270/01.04.01_60/ts_103270v010401p.pdf

## Scope
Defines the methodology for discovering an Authoritative FQDN for a radio service using DNS queries to radiodns.org. Covers construction of the RadioDNS FQDN, the unique ServiceIdentifier, and the bearerURI for broadcast services (FM/RDS/RBDS, DAB/DAB+, DRM, AMSS, IBOC), IP-streamed services, and SPI SI document-sourced services. Also defines client identification for connecting devices.

## Key Concepts
- **RadioDNS FQDN**: Internet domain constructed from broadcast parameters solely for querying DNS (e.g. `09580.c586.ce1.fm.radiodns.org`)
- **Authoritative FQDN**: The service provider's internet domain, returned as a CNAME record in response to a RadioDNS FQDN query
- **ServiceIdentifier**: String uniquely identifying a radio service within the scope of an Authoritative FQDN
- **bearerURI**: Unique identifier for the service used in SPI SI documents
- **GCC (Global Country Code)**: 3-char hex code = Country Code nibble + 2-char ECC; used in FQDN/ServiceIdentifier construction
- **Client identifier**: Non-unique string sent by a device to a service provider for personalisation
- **bearer**: Method of carriage of the service (FM, DAB, DRM, etc.)

## Data Structures

### FM/RDS/RBDS Parameters (Table 1)
| Field | Type | M/O | Notes |
|-------|------|-----|-------|
| gcc | 3-char hex | M | Global Country Code; derived from Country Code + ECC |
| pi | 4-char hex | M | RDS/RBDS Programme Identification code |
| frequency | 5-char string | M | In units of 100 kHz; leading zero for <100 MHz (e.g. 95.8 MHz → "09580") |

- **RadioDNS FQDN**: `<frequency>.<pi>.<gcc>.fm.radiodns.org`
- **ServiceIdentifier**: `fm/<gcc>/<pi>/<frequency>`
- **bearerURI**: `fm:<gcc>.<pi>.<frequency>` (frequency may be `*` to signify any frequency)

### DAB/DAB+ Parameters (Table 5)
| Field | Type | M/O | Notes |
|-------|------|-----|-------|
| gcc | 3-char hex | M | Global Country Code |
| eid | 4-char hex | M | Ensemble Identifier |
| sid | 4- or 8-char hex | M | Service Identifier |
| scids | 1-char hex | M | Service Component Identifier within Service |
| uatype | 3-char hex | M (data components) / O (audio) | User Application Type |

- **RadioDNS FQDN**: `[<uatype>.]<scids>.<sid>.<eid>.<gcc>.dab.radiodns.org`
- **ServiceIdentifier**: `dab/<gcc>/<eid>/<sid>/<scids>[/<uatype>]`
- **bearerURI**: `dab:<gcc>.<eid>.<sid>.<scids>[.<uatype>]`

### DRM Parameters (Table 9)
| Field | Type | M/O | Notes |
|-------|------|-----|-------|
| sid | 6-char hex | M | Globally unique Service Identifier; no region prefix needed |
| appdomain | 1-char hex | M (data) / O | Application domain |
| uatype | 3-char hex | M (data) / O | User application type |

- **RadioDNS FQDN**: `[<uatype>.<appdomain>]<sid>.drm.radiodns.org`
- **ServiceIdentifier**: `drm/<sid>[/<appdomain>/<uatype>]`
- **bearerURI**: `drm:<sid>[.<appdomain>.<uatype>]`

### AMSS Parameters (Table 13)
| Field | Type | M/O | Notes |
|-------|------|-----|-------|
| sid | 6-char hex | M | Globally unique Service Identifier |

- **RadioDNS FQDN**: `<sid>.amss.radiodns.org`
- **ServiceIdentifier**: `amss/<sid>`
- **bearerURI**: `amss:<sid>`

### IBOC Parameters (Table 14)
| Field | Type | M/O | Notes |
|-------|------|-----|-------|
| tx | 5-char hex | M | FCC facility code (US) or licensor-supplied unique ID (elsewhere) |
| cc | 3-char hex | M | Hex-encoded country ID; US = 0x292 |
| mId | 1-char hex | M (SPS) / O (main) | Multicast SPS channel ID; HD-2 = 2, HD-3 = 3, etc. Omitted for HD-1 |

- **RadioDNS FQDN**: `[<mId>.]<tx>.<cc>.hd.radiodns.org`
- **ServiceIdentifier**: `hd/<cc>/<tx>[/<mId>]`
- **bearerURI**: `hd:<cc>.<tx>[.<mId>]`

### IP-Streamed Service Parameters
| Field | Type | M/O | Notes |
|-------|------|-----|-------|
| fqdn | string | M | Authoritative FQDN sent in stream metadata |
| sid | ≤16 chars [a-z][0-9] | M | Unique per Authoritative FQDN |

- **bearerURI**: URL of the stream source (e.g. `http://media-ice.musicradio.com/Capital`)
- **ServiceIdentifier**: `id/<fqdn>/<sid>`

#### Streaming transport encodings:
| Transport | Method |
|-----------|--------|
| SHOUTcast | HTTP response header `icy-url: http://<fqdn>/<sid>` |
| ASF | Custom Metadata key/value: `radiodns-fqdn` and `radiodns-sid` (null-terminated Unicode) |
| Flash Audio | Read-only Remote Shared Object named `radiodns` with properties `fqdn` and `sid` |

### SPI SI Source
- `radiodns/@fqdn` → Authoritative FQDN
- `radiodns/@serviceIdentifier` → sid
- **ServiceIdentifier**: `id/<fqdn>/<sid>`
- Example: `<radiodns fqdn="www.heart.co.uk" serviceIdentifier="bristol"/>`

### Client Identifier
| Field | Constraint |
|-------|-----------|
| Format | 16–128 characters from `[a-zA-Z0-9]` |
| Scope | Bound to specific Authoritative FQDN(s) |
| Transport | HTTPS only, using HTTP Authorization header |
| Auth scheme | `ClientIdentifier` (custom scheme) |
| Credential syntax | token68 (RFC 7235 §2.1) containing the identifier string |

Example request header: `Authorization: ClientIdentifier ABCD123456`

## Rules and Constraints

### FQDN Resolution
1. [M] DNS query on RadioDNS FQDN SHALL return a single CNAME record containing the Authoritative FQDN. If no CNAME returned, service is not registered.
2. [M] Broadcast parameters SHALL be continuously monitored. If any parameter changes, FQDN resolution SHALL be repeated.
3. [M] The TTL of the Authoritative FQDN SHALL be queried and respected.
4. [M] Upon TTL expiry, FQDN resolution SHALL be repeated.
5. [M] If Authoritative FQDN changes, all active applications SHALL be notified and SHALL repeat their connection process.
6. [M] For IP-streamed services, if `fqdn` or `sid` values change, old values are expired and resolution SHALL be repeated.

### Metadata Intervals
7. [O] Service providers SHOULD ensure connecting clients receive `fqdn`/`sid` parameters within 5 seconds.

### GCC Construction
8. [M] GCC SHALL be constructed by concatenating the 1-char hex Country Code with the 2-char hex ECC.
9. [M] For FM/DAB audio: Country Code = first nibble of PI/SId; ECC from RDS Group 1A or DAB FIG 0/9.
10. [M] For DAB data services: Country Code = third nibble of DAB SId; ECC = first and second nibbles of DAB SId.
11. [O] If ECC not promptly received, receivers SHOULD use the GCC lookup table (Annex A, Table A.1) with ISO 3166-1 country code from GPS/GSM/GeoIP.

### Service Provider Implementation
12. [M] For FM/RDS/RBDS or DAB: service provider SHALL transmit ECC via RDS Group 1A or DAB FIG 0/9.
13. [M] For IBOC: service provider SHALL transmit Country Code and FCC Facility ID via Station ID Number message (MSG ID = 0000).
14. [M] For IP-streamed services: service provider SHALL provide `fqdn` and `sid` via at least one of clause 6 (stream metadata) or clause 7 (SPI SI).

### Device Implementation
15. [M] Device SHALL support clause 5 (broadcast FQDN resolution) for broadcast services.
16. [M] Device SHALL support acquisition of `fqdn` and `sid` from at least one of clause 6 or clause 7 for IP-streamed services.

### Client Identification
17. [M] Client identifiers SHALL be kept private; SHALL NOT be exposed via UI or external connections.
18. [M] Client SHALL only present identifier associated with the Authoritative FQDN of the application being requested.
19. [M] Client identifier presentation SHALL use TLS encryption.
20. [M] Client SHALL present identifier using `Authorization: ClientIdentifier <token>` HTTP header.
21. [M] Service provider SHALL return HTTP 200 for valid identifier, HTTP 401 (with `WWW-Authenticate: ClientIdentifier`) when no identifier presented, HTTP 403 for invalid/unrecognized identifier.
22. [M] On receiving HTTP 403, client SHALL NOT repeat the same request.
23. [M] Service provider SHALL always return a valid response if no identifier is provided.
24. [O] Authentication parameter `realm` is optional in challenges; `charset` is optional.

## Workflows and Sequences

### Broadcast Service FQDN Discovery
1. [M] Extract broadcast parameters (gcc, pi/eid/sid/etc.) from received signal
2. [M] Construct RadioDNS FQDN from parameters per bearer type
3. [M] Perform DNS query on RadioDNS FQDN
4. [M] Retrieve CNAME record → Authoritative FQDN
5. [M] Note TTL; schedule re-resolution on expiry
6. [M] Proceed to application discovery using Authoritative FQDN (per application spec)

### IP-Streamed Service FQDN Discovery (Clause 6)
1. [M] Connect to stream
2. [M] Read `fqdn` and `sid` from in-stream metadata (SHOUTcast/ASF/Flash)
3. [M] Monitor for parameter changes; re-resolve if changed
4. [M] Construct ServiceIdentifier as `id/<fqdn>/<sid>`

### SPI SI FQDN Discovery (Clause 7)
1. [O] Acquire SPI SI document per ETSI TS 102 818
2. [M] Read `radiodns/@fqdn` and `radiodns/@serviceIdentifier`
3. [M] Discard parameters if SI expires; repeat resolution
4. [M] Construct ServiceIdentifier as `id/<fqdn>/<sid>`

### Client Identifier Flow (HTTPS)
1. [O] Check application SRV records to determine if client identification is supported
2. [O] Present `Authorization: ClientIdentifier <token>` header in HTTPS request
3. [M] Handle response: 200 (success), 401 (no identifier → provide one), 403 (invalid → do not retry)

## Conformance Requirements
- Implement RadioDNS FQDN construction for all applicable bearer types (FM, DAB, DRM, AMSS, IBOC)
- Perform DNS CNAME lookup and extract Authoritative FQDN
- Respect DNS TTL; re-resolve on expiry or parameter change
- For FM/DAB: transmit/receive ECC; for IBOC: transmit/receive Country Code + tx ID
- Support at least one FQDN source for IP-streamed services (stream metadata or SPI SI)
- Client identifier transport uses HTTPS with `ClientIdentifier` auth scheme
- Service provider returns correct HTTP status codes for client identifier scenarios
- Client identifiers must not be exposed beyond the client

## Summary Notes
- The GCC fallback table (Annex A, Table A.1) covers all ITU-registered countries with bordering country lists for cross-border reception scenarios. Countries where no RDS/DAB codes are allocated (marked X/XX) register under an adjoining country.
- Table A.2 lists country pairs explicitly excluded as bordering due to >100–200 km sea separation.
- The Canada/US edge case: RDS Country Code B is shared; PI codes B1 and B8–BF → US; B2–B7 → Canada.
- The `bearerURI` frequency field for FM may use `*` wildcard when the exact frequency is unknown, relying on PI code alone for service location.
- Client identifier allocation, distribution, and management by service providers is outside the scope of this document.
- The standard does not define which specific SRV record signals client identification support — that is delegated to individual application specifications (e.g. RadioVIS, RadioEPG, RadioTag).
- V1.4.1 (2022-05) adds client identification (clause 9) relative to earlier versions.
